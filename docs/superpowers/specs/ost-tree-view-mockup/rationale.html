OST Tree View — design rationale

One artifact, two cuts of the same tree. A breadth view for choosing an opportunity, a depth view for working the chosen branch. Built to port straight into a dependency-free index.html.

1 · Layout & the two cuts

Opportunity space (breadth)

A top-down tree: outcome at the root, the full opportunity space branching beneath it (up to two levels). It stops at the opportunity layer — no solutions, no table. The chosen opportunity gets the blue ring + Vald badge; the rest are de-emphasised (lower opacity) so they read as "the space not taken."

Used while deciding which opportunity to pursue.

Chosen branch (depth)

A narrow vertical slice: outcome → chosen opportunity → top-3 solutions → the assumption table spanning the bottom. We deliberately don't draw every opportunity's sub-tree at once — that's unreadable — so depth lives on one branch.

Used once an opportunity is chosen.

The two share one tree tab, switched by a segmented toggle (Opportunity space ⇄ Chosen branch) in the Scilla pill style. The toggle re-renders the visible cut from the same data object, so there is a single source of truth.

2 · Swimlane bands

Bands are full-width tinted rows behind the tree — PRODUCT OUTCOME (yellow), MÖJLIGHETER (teal), LÖSNINGAR (cream); the breadth cut shows only the first two. Labels are small italic uppercase in a sticky left gutter, so they stay pinned while the tree scrolls horizontally.

Crucially, band heights are measured from the DOM after render (each node is tagged data-band; the boundary sits at the mid-point between one band's lowest node and the next band's highest). This means bands always track real content — they don't drift if a node wraps to more lines, and they adapt automatically to empty/partial states. No hard-coded pixel heights.

3 · Nodes & edges

Nodes

  • Outcome — filled yellow lozenge (rounded-rect), the visual anchor.
  • Opportunity — outlined teal pill. Chosen = blue border + brutalist blue shadow + badge.
  • Solution — small cream pill, numbered 1.1 / 1.2 / 1.3 (the same id used in the table's Lösning column, so tree and table cross-reference cleanly).

Edges

Orthogonal (right-angle) connectors via the canonical pure-CSS tree: each <li> draws its half of the horizontal bus with ::before/::after borders; each child <ul> draws the vertical down-stem. A single child collapses to a straight line. Arrowheads are a CSS triangle (border) on each node's top — no SVG file, no measurement, no library.

4 · The assumption table (ANTAGANDEN)

A dense, spreadsheet-style table, grouped by solution (each group has a header row + count). Filled columns: Antagande, Lösning, Risknivå, Typ av antagande, Testmetod, Framgångskriterier. The three execution-time fields — Status, Issue ID, Insikter — are rendered as greyed, hatched columns labelled "(coming soon)" so the eventual shape is visible without faking data.

Riskiest assumptions (High importance + weak evidence → Risknivå: High) are flagged with a red left rule on the row and a coral risk chip. Type chips are colour-coded to the five categories (a legend sits above the table).

5 · Empty / partial states

The renderer is data-driven, so missing layers degrade gracefully. A demo control on the depth cut (Full · No assumptions · No solutions) lets you see all three:

  • No solutions → the LÖSNINGAR band shows a dashed placeholder node ("Inga lösningar ännu…") on a dashed stub; the table shows a "choose a solution first" empty state.
  • No assumptions → solutions still render; the table shows an empty state with a kaomoji.
  • Per-group empty (always on in Full) → solution 1.3 has no assumptions yet, so its group shows an inline "inga antaganden ännu" row — partial data within an otherwise full table.

6 · Responsive & scroll behaviour

  • The tree lives in a horizontally scrollable canvas; a wide opportunity space scrolls rather than shrinking. On load the canvas auto-centres on the root outcome so the tree reads top-down.
  • The band gutter narrows on small screens (172px → 118px) and band sub-labels hide under 640px.
  • The table scrolls horizontally inside its own card (min-width: 1120px) so cells never crush; the page itself stays single-column.

7 · Colour tokens

Everything theme-able is a --ost-* custom property at :root, mapped onto Scilla --sc-* values. Remap these to retheme:

Open questions & trade-offs

How deep can opportunity nesting go? The pure-CSS tree handles arbitrary depth, but the breadth cut gets unreadable past ~2 levels (the brief's cap) — beyond that, sibling pills collide horizontally and the row grows very wide. Recommendation: keep the cap at 2 and, for deeper structure, push users into the depth cut. If 3+ levels are ever required, we'd want collapse/expand on opportunity nodes.
The table on narrow screens. Today it scrolls horizontally inside its card with a 1120px floor. That preserves the spreadsheet feel but means mobile users scroll sideways. Alternative for true mobile: collapse each row into a stacked card. Worth deciding who actually opens this on a phone.
Centring a very wide breadth tree. The root centres over the whole opportunity span, so with many top-level opportunities the outcome sits well right of the first one. We auto-scroll to frame the root on load, but for 6+ opportunities a mini-map or "fit" control would help.
Solution numbering. The brief mentioned both 1.1… (nodes) and S1… (table). We unified on 1.x everywhere so the tree and table share one identifier — flag if you'd rather keep S1.
Sub-opportunities in the depth cut. We render outcome → chosen opportunity → solutions directly (solutions hang off the parent opportunity per the data). The chosen opportunity's own sub-opportunities (O1a/O1b) are shown only in the breadth cut to avoid an ambiguous "do solutions attach to the parent or the children?" picture. Easy to add as a context row if wanted.
